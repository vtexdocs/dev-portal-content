In this guide, you'll learn how to migrate your storefront from [Store Framework](https://developers.vtex.com/docs/guides/store-framework) to [FastStore](https://developers.vtex.com/docs/guides/faststore).

By the end, you'll know how to prepare both storefronts to run in parallel, validate the new storefront end-to-end and with real traffic, and complete the cutover.

> ℹ️ You can choose between two approaches for validating FastStore before go-live: Develop and test directly in your production VTEX account using a temporary second domain, or in an isolated QA subaccount. To learn more about both approaches and their trade-offs, see [Choose your test environment](#choose-your-test-environment) in the Before you begin section. The final cutover always happens on your production account's domain.

## Migration overview

The migration runs both storefronts in parallel. Your Store Framework storefront stays live while you develop, validate, and gradually roll out the FastStore storefront.

In the following table, see the role of each storefront during the migration:

| Stage | Store Framework (current storefront) | FastStore (new storefront) |
| ----- | ------------------------------------ | ----------------------------- |
| Development | Remains live. No changes are made. | Develop the FastStore project and connect the repository to [FastStore WebOps](https://developers.vtex.com/docs/guides/faststore/webops-dashboard). For more details, see the [Before you begin](#before-you-begin) section. |
| Test environment | Not affected. | Decide whether to test in a second domain in the same account or on a QA subaccount. For more details, see [Choosing your test environment](#choosing-your-test-environment). |
| VTEX IO apps | Map installed apps. | Backend apps are fully compatible: nothing to do if migrating in the same account, or reinstall them if using a QA subaccount. Front-end components aren't compatible and must be recreated. For more details, see [Step 1 - Handling VTEX IO apps](#step-1-handling-vtex-io-apps). |
| Content | [Site Editor](https://help.vtex.com/docs/tutorials/site-editor-overview) content remains as is. | Recreate storefront content by using [CMS for FastStore storefronts](https://developers.vtex.com/docs/guides/cms-for-faststore-storefronts). For more details, see [Step 2 - Recreating the storefront content](#step-2-recreating-the-storefront-content). |
| Front-end third-party tools | Remain as is. | Reimplement front-end tools (analytics, consent, heatmaps, etc.) and validate them during end-to-end testing. For more details, see [Step 3 - Reimplementing front-end third-party tools](#step-3-reimplementing-front-end-third-party-tools). |
| End-to-end validation | Keeps serving live traffic. | Validate the full shopper journey before requesting traffic. For more details, see [Step 4 - Validating end-to-end flows](#step-4-validating-end-to-end-flows). |
| Production domain | Keeps serving the live production domain. | Add the production domain to FastStore WebOps so FastStore is ready for cutover, but it still doesn't receive live traffic. For more details, see [Step 5 - Adding the production domain to WebOps](#step-5-adding-the-production-domain-to-webops). |
| Traffic splitting | Shares live traffic with FastStore during the transition. | Gradually receives traffic, starting at 0%, until validated. For more details, see [Step 6 - Requesting the traffic split](#step-6-requesting-the-traffic-split). |
| Completion | Retired after a successful cutover. | Becomes the live storefront. |

## Before you begin

<Steps>

### Check your Edition App version

Confirm the Edition App of your Store Framework account is set to `vtex.edition-store@5.x` in your master workspace:

1. Using the [VTEX IO CLI](https://developers.vtex.com/docs/guides/vtex-io-documentation-vtex-io-cli-installation-and-command-reference), log in to your account by running the `vtex login {accountName}` command. Replace `{accountName}` with the name of your account.
2. Check the installed Edition App version by running `vtex edition get`.

If the result is `vtex.edition-store@5.x`, no action is needed. If it's an older major, open a ticket with [VTEX Support](https://help.vtex.com/en/support) to request the installation of `vtex.edition-store@5.x`, which FastStore requires.

> ❗ Changing the Edition App is an account-wide operation that applies to the master workspace and may update the major versions of some base apps. Since your Store Framework storefront is live, align the timing and post-change validation with VTEX Support when you request the upgrade. For more details about the version differences and the impact on dependencies, see [Edition App](https://developers.vtex.com/docs/guides/vtex-io-documentation-edition-app).

### Choose your test environment

Because traffic splitting only happens once development is complete, you need a running production-like domain on FastStore earlier, to validate specific flows that can't be tested from preview URLs alone, such as login-gated features or the transition from the minicart to checkout.

There are two supported ways to get that test domain during development:

- **Same account:** Keep your existing domain pointed at Store Framework, and add a second domain in the same account pointed at FastStore (for example, `www.yourstore.com` for Store Framework and `faststore.yourstore.com` for FastStore). This keeps all development and testing — Master Data, CMS content, app settings — in a single account. This second domain is only a temporary stand-in for testing during development. It's separate from your actual production domain, which you add to FastStore WebOps later in [step 5](#step-5-adding-the-production-domain-to-webops), and which is the only domain involved in the real go-live and traffic split.
- **QA subaccount:** Create a separate subaccount, connect your FastStore project to it, and point a test domain there (for example, `faststoreqa.yourstore.com` hosted on a subaccount). This isolates your tests behind a completely separate Master Data instance. This path is useful mainly when you need an isolated environment for destructive tests or order-flow testing that shouldn't run against your production account.

The following table summarizes the trade-offs:

| Criteria | Same account | QA subaccount |
|---|---|---|
| Setup complexity | Lower: no new account, no content resync before go-live. You create a new FastStore project, but keep development and content in the existing production account. | Higher: requires a new QA subaccount, plus CMS content and app settings must be recreated in the main account later. |
| CMS content | Registered once using [CMS for FastStore storefronts](https://developers.vtex.com/docs/guides/cms-for-faststore-storefronts), directly in the production account. | Registered in the subaccount CMS for testing, then it must be recreated a second time in the production account before go-live, since subaccount CMS content isn't copied over automatically. |
| Master Data | Shared with production (use with care for tests that write data). | Fully isolated, safe for destructive tests. |
| Order simulation | Because the checkout is unchanged, placing real orders is usually unnecessary for storefront validation. Any test orders you do place run in your production account context and are subject to the same billing rules as real orders. | Lets you place test orders in a fully isolated context when you need to exercise order flows without touching production. Choose the QA subaccount path mainly when you need an isolated environment for destructive tests or order-flow testing that shouldn't run against your production account. |
| Intelligent Search | Reuses existing account-level configuration; no rebuild needed. | Requires reconfiguring merchandising rules, synonyms, and relevance settings, since a subaccount is a separate environment. |
| Best fit | Teams that want the fastest path to production and prefer to keep development, content, and account-level configuration in the same VTEX account. | Teams that need stronger isolation for testing, already operate with separate accounts or subaccounts, or want a reusable QA environment after go-live. |
| Post-go-live use | None by default (the temporary test domain is retired once the production domain takes over). | Reusable afterward as a standing QA environment. |

### Create your FastStore Project

Before starting the migration, create the FastStore project that will receive your storefront development. To start one, follow the guides in the FastStore [Getting started](https://developers.vtex.com/docs/guides/faststore/getting-started-overview) track.

Once you complete the track, you'll have a GitHub repository connected to your account that you can deploy through FastStore WebOps. If you go with the same-account approach, you connect the repository directly to your main account and avoid any later repository re-pointing or content resync work.

### Set your testing domain

If you choose the same-account approach and need a production-like domain for end-to-end validation, set up the temporary FastStore domain at this stage. Configure the subdomain in your DNS provider, add it to VTEX Account Settings, then add it to FastStore WebOps. For detailed instructions, see [Configuring external DNS](https://developers.vtex.com/docs/guides/faststore/go-live-1-configuring-external-dns).

If you chose the QA subaccount approach instead, use the domain already pointed to that subaccount for these validations. If you need to create a new subaccount, see [Managing a multistore](https://help.vtex.com/docs/tutorials/managing-a-multistore), then follow the instructions in the Configuring external DNS guide mentioned above.

</Steps>

## Instructions

### Step 1 - Handling VTEX IO apps

Backend VTEX IO apps and integrations are fully compatible with FastStore. If you migrate within the same account, no action is needed — your existing backend apps keep running as is. If you migrate using a QA subaccount, reinstall the same backend apps in that subaccount before you start development.

Front-end VTEX IO components, on the other hand, aren't compatible with FastStore at all and must be recreated regardless of which account you use. Start by checking the [FastStore UI](https://developers.vtex.com/docs/guides/faststore/components-index) catalog to see if a native component already covers your needs. If it doesn't, see [Developing and overriding UI components](https://developers.vtex.com/docs/guides/faststore/developing-and-overriding-components-overview) to learn how to override native components or build entirely new sections to recreate your custom front-end elements.

### Step 2 - Recreating the storefront content

Recreate all necessary pages, components, and content in the [CMS for FastStore storefronts](https://developers.vtex.com/docs/guides/cms-for-faststore-storefronts).

> ℹ️ Content migration between Store Framework Site Editor and FastStore CMS isn't supported, so plan enough time to recreate and review the content manually.

If you migrate within the same account, you register this content once, directly in the account that will go live. If you use a QA subaccount instead, plan for an additional step to recreate this content into the main account before cutover.

### Step 3 - Reimplementing front-end third-party tools

Because the FastStore front-end code is independent of the Store Framework, any third-party tools that run on the front-end must be reimplemented in your FastStore project to ensure they continue working after migration. This typically includes analytics, consent management, and behavior-tracking tools (for example, Google Tag Manager, Hotjar, and similar platforms).

Keep these tools maintained throughout your FastStore development, and validate them as part of the end-to-end flows described in [step 4](#step-4-validating-end-to-end-flows). Before requesting the traffic split, confirm that all reimplemented tools fire correctly on the FastStore front-end, so you have a reliable baseline for comparison once traffic starts flowing to both variants.

### Step 4 - Validating end-to-end flows

Validate the full shopper journey on FastStore: Catalog browsing, adding products to the cart, cart and checkout, login, and My Account. Most of this can be reviewed from preview URLs, but a handful of flows, such as login-gated features or the minicart-to-checkout transition, require a live production domain running FastStore.

Use whichever production-like domain you set up in [Choosing your test environment](#choosing-your-test-environment) to run these tests, either the second domain in your main account or your QA subaccount domain.

Keep the following in mind when performing end-to-end tests:

- **Checkout:** If your checkout remains unchanged, focus this validation on the storefront-to-checkout transition and related authenticated flows. Placing real orders is usually unnecessary unless you specifically need to validate order creation behavior in the chosen environment.
- **Customized flows:** Because both storefronts share the same production domain during the traffic split, login redirects and links into the checkout panel generally continue to work without extra configuration, even if you have login or checkout UI customizations. Still, verify these flows in your test environment before requesting the split, since Store Framework allows extensive customization that can behave differently on a case-by-case basis.
- **Header and footer:** Because the header and footer are shared between Store Framework and FastStore during the traffic split, verify that all header and footer links and experiences work consistently on both storefront versions before requesting the split. If your store has URLs unique to a single storefront, you may need to set up redirects to keep navigation consistent for shoppers across both versions. For instructions, see [Managing URL redirects](https://developers.vtex.com/docs/guides/vtex-io-documentation-managing-url-redirects).
- **Third-party tools:** Confirm that all front-end third-party tools you reimplemented in the [previous step](#step-3-reimplementing-front-end-third-party-tools) are firing correctly on the FastStore domain.
- **Master Data:** If you use a QA subaccount, it has its own Master Data, so the data used in tests is isolated from your live store. If you're testing on a second domain in the main account, be mindful that Master Data is shared with production.

### Step 5 - Adding the production domain to WebOps

> ℹ️ [DNS configuration on VTEX](https://help.vtex.com/en/docs/tutorials/how-dns-configuration-works-on-vtex) points to the platform in general and isn't exclusive to Store Framework or FastStore. Since your production domain already works on Store Framework, you don't need to configure a new domain or subdomain for the traffic split. You only need to register your existing production domain in FastStore WebOps, as described below, and share this context with [VTEX Support](https://help.vtex.com/en/support) when you request the traffic split in [step 6](#step-6-requesting-the-traffic-split), so they can confirm everything is set up as expected.

Before requesting the traffic split, add your production domain to FastStore WebOps by following these steps:

1. In your VTEX Admin, go to **Storefront > FastStore WebOps**.
2. Navigate to the **Settings** tab.
3. In the **Production domain** card, select your production domain from the dropdown list under the **Host** section.
4. Click `Add`.

    ![adding-host-webops](https://cdn.jsdelivr.net/gh/vtexdocs/dev-portal-content@main/docs/faststore/docs/go-live/adding-host-webops.png)

> ℹ️ Adding the domain doesn't change your live storefront, which keeps pointing to Store Framework until you request a traffic split.

### Step 6 - Requesting the traffic split

> ⚠️ Before requesting the traffic split, make sure your FastStore storefront is fully ready to go live. If you developed on a second domain in the same account, confirm the production domain was added to WebOps in [step 5](#step-5-adding-the-production-domain-to-webops). If you developed on a QA subaccount, make sure you've also recreated the CMS content and reinstalled the necessary apps in your main account, since none of that is carried over automatically.

Once you've completed the previous steps, open a ticket with [VTEX Support](https://help.vtex.com/en/support) at least five days before your planned migration date to request traffic splitting between Store Framework and FastStore.

When the split is enabled, we recommend starting with 0% of traffic routed to FastStore and 100% kept on Store Framework for final validations on the production domain. After confirming that routing, analytics, and critical flows are working as expected, gradually increase the FastStore share. This gradual rollout allows you to slowly redirect users from the old store to the new one, run A/B tests with your analytics or testing tools, monitor performance, and ensure stability before going fully live on FastStore.

After the split is active, you can force each storefront version for QA and debugging by adding the `ab_test` query parameter to the URL:

- To force the Store Framework version: `https://www.store.com/?ab_test=v1`
- To force the FastStore version: `https://www.store.com/?ab_test=v2`

To keep the shopper experience consistent, VTEX persists the assigned storefront version in the `VtexStoreVersion` cookie, so the same shopper stays on the same variant rather than being re-randomized between the two versions on subsequent requests.

You can also read the `VtexStoreVersion` cookie in tools such as Google Tag Manager (GTM) or Google Analytics (GA) to compare metrics across the two storefront variants.

> ℹ️ By default, the `VtexStoreVersion` cookie is set as `HttpOnly`, which prevents client-side tools like GTM from reading it. If you need to read it in the browser for analytics, request this adjustment when you open the traffic split ticket.

Once you've validated the FastStore storefront, make it your live store and retire the Store Framework storefront.
