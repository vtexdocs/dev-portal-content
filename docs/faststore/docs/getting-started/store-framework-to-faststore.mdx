In this guide, you'll learn how to migrate your storefront from [Store Framework](https://developers.vtex.com/docs/guides/store-framework) to [FastStore](https://developers.vtex.com/docs/guides/faststore) using a [subaccount](https://help.vtex.com/docs/tutorials/managing-a-multistore).

By the end, you'll know how to prepare both storefronts to run in parallel, validate the new storefront end-to-end and with real traffic, and complete the cutover.

> ℹ️ You'll develop and validate FastStore before going live in an isolated QA subaccount. To learn more about this test environment and what it entails, see [Set up your test environment](#set-up-your-test-environment) in the Before you begin section. The final cutover always happens on your production account's domain.

## Migration overview

The migration runs both storefronts in parallel. Your Store Framework storefront stays live while you develop, validate, and gradually roll out the FastStore storefront.

In the following table, see the role of each storefront during the migration:

| Stage | Store Framework (current storefront) | FastStore (new storefront) |
| ----- | ------------------------------------ | ----------------------------- |
| Test environment | Not affected. | Set up an isolated QA subaccount for development and testing. For more details, see [Set up your test environment](#set-up-your-test-environment). |
| Edition App | Check the Edition App version and upgrade to `vtex.edition-store@5.x` if it's an older major. | FastStore requires `vtex.edition-store@5.x`. For more details, see [Step 1 - Check your Edition App version](#step-1-check-your-edition-app-version). |
| Development | Remains live. No changes are made. | Develop the FastStore project and connect the repository to [FastStore WebOps](https://developers.vtex.com/docs/guides/faststore/webops-dashboard). For more details, see [Step 2 - Create your FastStore project](#step-2-create-your-faststore-project). |
| Testing domain | Keeps serving the live domain. | Stand up a production-like test domain on the QA subaccount: Configure the subdomain in your DNS provider, add it to VTEX Account Settings, then to FastStore WebOps. For more details, see [Step 3 - Set up your testing domain](#step-3-set-up-your-testing-domain). |
| VTEX IO apps | Map installed apps. | Backend apps are fully compatible: reinstall them in the QA subaccount. Front-end components aren't compatible and must be recreated. For more details, see [Step 4 - Handling VTEX IO apps](#step-4-handling-vtex-io-apps). |
| Content | [Site Editor](https://help.vtex.com/docs/tutorials/site-editor-overview) content remains as is. | Recreate storefront content by using [CMS for FastStore storefronts](https://developers.vtex.com/docs/guides/cms-for-faststore-storefronts). For more details, see [Step 5 - Recreating the storefront content](#step-5-recreating-the-storefront-content). |
| Front-end third-party tools | Remain as is. | Reimplement front-end tools (analytics, consent, heatmaps, etc.) and validate them during end-to-end testing. For more details, see [Step 6 - Reimplementing front-end third-party tools](#step-6-reimplementing-front-end-third-party-tools). |
| End-to-end validation | Keeps serving live traffic. | Validate the full shopper journey before requesting traffic. For more details, see [Step 7 - Validating end-to-end flows](#step-7-validating-end-to-end-flows). |
| Production account setup | Remains live. Backend apps and Intelligent Search are already in place. | Recreate the FastStore-specific setup in your production account. For more details, see [Step 8 - Setting up FastStore in your production account](#step-8-setting-up-faststore-in-your-production-account). |
| Production domain | Keeps serving the live production domain. | Add the production domain to FastStore WebOps so FastStore is ready for cutover, but it still doesn't receive live traffic. For more details, see [Step 9 - Adding the production domain to WebOps](#step-9-adding-the-production-domain-to-webops). |
| Traffic splitting | Shares live traffic with FastStore during the transition. | Gradually receives traffic, starting at 0%, until validated. For more details, see [Step 10 - Requesting the traffic split](#step-10-requesting-the-traffic-split). |
| Completion | Retired after a successful cutover. | Becomes the live storefront. |

## Before you begin

### Set up your test environment

Because traffic splitting only happens once development is complete, you need a running production-like domain on FastStore earlier, to validate specific flows that can't be tested from preview URLs alone, such as login-gated features or the transition from the minicart to checkout.

To get that test domain during development, use an isolated QA subaccount: Create a separate subaccount, connect your FastStore project to it, and point the test domain to it (for example, `faststoreqa.yourstore.com` hosted on that subaccount). This isolates your tests behind a completely separate Master Data instance, which is safe for destructive tests and order-flow testing that shouldn't run against your production account. The subaccount can also be reused afterward as a standing QA environment.

If you still need to create the subaccount, see [Creating a multistore](https://help.vtex.com/docs/tutorials/managing-a-multistore#creating-a-multistore).

Keep the following in mind when working with a QA subaccount:

- **CMS content:** Content you register in the subaccount CMS is for testing only. It must be recreated a second time in the production account before go-live, since subaccount CMS content isn't copied over automatically. For more information, see [step 8](#step-8-setting-up-faststore-in-your-production-account).
- **Master Data:** Fully isolated from production, so it's safe for destructive tests and for placing test orders in a separate context.
- **Intelligent Search:** Requires reconfiguring merchandising rules, synonyms, and relevance settings, since a subaccount is a separate environment.
- **VTEX IO apps:** Backend apps and integrations must be reinstalled in the subaccount before you start development.
- **Edition App:** FastStore requires `vtex.edition-store@5.x`. Confirm your QA subaccount is set to this major before you start development, using the same procedure described in [Step 1 - Check your Edition App version](#step-1-check-your-edition-app-version). If it's an older major, open a ticket with [VTEX Support](https://help.vtex.com/en/support) to upgrade the subaccount.

> ℹ️ Because development and content live in the subaccount while the cutover happens on your production account, plan for a step to connect the FastStore project and recreate the CMS content in the production account before requesting the traffic split. Backend apps and Intelligent Search are already in place there, as they are in your live storefront.

## Instructions

### Step 1 - Check your Edition App versions

FastStore requires `vtex.edition-store@5.x`, so validate which [Edition App](https://developers.vtex.com/docs/guides/vtex-io-documentation-edition-app) version is installed in both your production account and QA subaccount. This verification process guarantees that your environments meet the necessary requirements for proper FastStore operation.

#### Check the QA subaccount

Check that the Edition App of your QA subaccount is set to `vtex.edition-store@5.x` in the master workspace:

1. Using the [VTEX IO CLI](https://developers.vtex.com/docs/guides/vtex-io-documentation-vtex-io-cli-installation-and-command-reference), log in to your account by running the `vtex login {qaAccountName}` command. Replace `{qaAccountName}` with the subaccount name.
2. Check the installed Edition App version by running `vtex edition get`.

If the result is `vtex.edition-store@5.x`, continue to [step 2](#step-2-create-your-faststore-project).

If the result shows an older major version, open a ticket with [VTEX Support](https://help.vtex.com/en/support) requesting the installation of `vtex.edition-store@5.x` in the QA subaccount's master workspace. Wait for the upgrade to be completed before creating the FastStore project.

#### Check the production account

Check that the Edition App of your production account, where your Store Framework storefront lives, is set to `vtex.edition-store@5.x` in the master workspace:

1. Using the [VTEX IO CLI](https://developers.vtex.com/docs/guides/vtex-io-documentation-vtex-io-cli-installation-and-command-reference), log in to your account by running the `vtex login {accountName}` command. Replace `{accountName}` with the account name.
2. Check the installed Edition App version by running `vtex edition get`.

If the result is `vtex.edition-store@5.x`, no upgrade is required. If it shows an older major version, plan the upgrade as part of [Step 8 - Setting up FastStore in your production account](#step-8-setting-up-faststore-in-your-production-account). Don't connect the FastStore project to the production account until the upgrade is complete.

> ⚠️ Changing the Edition App is an account-wide operation that applies to the master workspace and may update the major versions of some base apps. Since your Store Framework storefront is live, align the timing and post-change validation with VTEX Support when you request the upgrade. For more details on version differences and their impact on dependencies, see [Edition App](https://developers.vtex.com/docs/guides/vtex-io-documentation-edition-app).

### Step 2 - Create your FastStore project

Create the FastStore project that will receive your storefront development. To start one, follow the guides in the FastStore [Getting started](https://developers.vtex.com/docs/guides/faststore/getting-started-overview) track.

Once you complete the track, you'll have a GitHub repository connected to your QA subaccount that you can deploy through FastStore WebOps.

### Step 3 - Set up your testing domain

Set up a production-like domain on your QA subaccount for end-to-end validation: Configure the subdomain in your DNS provider, add it to VTEX account settings, then add it to FastStore WebOps. At the end, you need to open a ticket with [VTEX Support](https://help.vtex.com/en/support) requesting a domain migration for your FastStore WebOps.

For the complete instructions, see [Configuring external DNS](https://developers.vtex.com/docs/guides/faststore/go-live-1-configuring-external-dns).

> ℹ️ You don't need to configure an external DNS for the production domain, since this was already done when your Store Framework store went live. For more information, see [step 9](#step-9-adding-the-production-domain-to-webops).

### Step 4 - Handling VTEX IO apps

Start by taking an inventory of the VTEX IO apps installed in your Store Framework account, so you know which ones to reinstall or recreate in your QA subaccount.

Backend VTEX IO apps and integrations are fully compatible with FastStore. Reinstall the same backend apps in your QA subaccount before you start development.

Front-end VTEX IO components, on the other hand, aren't compatible with FastStore at all and must be recreated. Start by checking the [FastStore UI](https://developers.vtex.com/docs/guides/faststore/components-index) catalog to see if a native component already covers your needs. If it doesn't, see [Developing and overriding UI components](https://developers.vtex.com/docs/guides/faststore/developing-and-overriding-components-overview) to learn how to override native components or build entirely new sections to recreate your custom front-end elements.

### Step 5 - Recreating the storefront content

Recreate all necessary pages, components, and content in the [CMS for FastStore storefronts](https://developers.vtex.com/docs/guides/cms-for-faststore-storefronts).

> ℹ️ Content migration between Store Framework Site Editor and FastStore CMS isn't supported, so plan enough time to recreate and review the content manually.

Because you develop in a QA subaccount, plan for an additional step to recreate this content in the main account before cutover, since subaccount CMS content isn't carried over automatically.

### Step 6 - Reimplementing front-end third-party tools

Because the FastStore front-end code is independent of the Store Framework, any third-party tools that run on the front-end must be reimplemented in your FastStore project to ensure they continue working after migration. This typically includes analytics, consent management, and behavior-tracking tools (for example, Google Tag Manager, Hotjar, and similar platforms).

Keep these tools maintained throughout your FastStore development, and validate them as part of the end-to-end flows described in [step 7](#step-7-validating-end-to-end-flows). Before requesting the traffic split, confirm that all reimplemented tools fire correctly on the FastStore front-end, so you have a reliable baseline for comparison once traffic starts flowing to both variants.

### Step 7 - Validating end-to-end flows

Validate the full shopper journey on FastStore: Catalog browsing, adding products to the cart, cart and checkout, login, and My Account. Most of this can be reviewed from preview URLs, but a handful of flows, such as login-gated features or the minicart-to-checkout transition, require FastStore to run on a production-like domain.

Use the production-like domain you set up on your QA subaccount in [Step 3 - Set up your testing domain](#step-3-set-up-your-testing-domain) to run these tests.

Keep the following in mind when performing end-to-end tests:

- **Checkout and order placement:** If your checkout remains unchanged, focus this validation on the storefront-to-checkout transition and related authenticated flows. Placing real orders is usually unnecessary unless you specifically need to validate order creation behavior in the chosen environment.
- **Customized flows:** Because both storefronts share the same production domain during the traffic split, login redirects and links into the checkout panel generally continue to work without extra configuration, even if you have login or checkout UI customizations. Still, verify these flows in your test environment before requesting the split, since Store Framework allows extensive customization that can behave differently on a case-by-case basis.
- **Checkout, Order Placed, and My Account navigation:** Checkout, Order Placed, and My Account use the existing VTEX IO theme and are shared by shoppers routed to either Store Framework or FastStore during the traffic split. Verify that their header, footer, and other navigation links point to URLs available in both storefronts. Prefer using the same URLs in both versions. When equivalent pages use different URLs, configure the redirect in the storefront where the original URL isn't available. See [Managing URL redirects](https://developers.vtex.com/docs/guides/vtex-io-documentation-managing-url-redirects) for Store Framework and [Managing URLs with redirects and rewrite paths](https://developers.vtex.com/docs/guides/faststore/routing-managing-urls-with-redirects-and-rewrite-paths) for FastStore.

    > ⚠️ To avoid a redirect loop, redirects that send a FastStore route to Store Framework through `/api/io/{path}` must be configured in FastStore, not through Store Framework Admin redirects.

- **Third-party tools:** Confirm that all front-end third-party tools you reimplemented in the [previous step](#step-6-reimplementing-front-end-third-party-tools) are firing correctly on the FastStore domain.
- **Master Data:** Your QA subaccount has its own Master Data, so the data used in tests is isolated from your live store.

### Step 8 - Setting up FastStore in your production account

Your production account is your live Store Framework store, so its backend VTEX IO apps, integrations, and Intelligent Search configuration are already in place and shared with FastStore. You don't need to reinstall or reconfigure them. What's missing on production is the FastStore-specific setup that lived only in your QA subaccount: the connected project (code) and the FastStore CMS content.

The code isn't recreated. During FastStore onboarding, you can connect to the same project you developed for end-to-end tests. However, FastStore CMS content must be recreated, as it cannot be migrated automatically.

While your Store Framework storefront keeps serving live traffic, complete the following in your production account:

1. **Upgrade the Edition App:** Confirm your production account is set to `vtex.edition-store@5.x`, as described in [Step 1 - Check your Edition App version](#step-1-check-your-edition-app-version). Since this is an account-wide operation on a live storefront, align the timing and post-change validation with [VTEX Support](https://help.vtex.com/en/support).
2. **Connect the FastStore project:** Connect the same GitHub repository you developed in [Step 2 - Create your FastStore project](#step-2-create-your-faststore-project) to your production account through FastStore WebOps, so it can be deployed on production.
3. **Recreate CMS content:** Recreate all pages, components, and content in the [CMS for FastStore storefronts](https://developers.vtex.com/docs/guides/cms-for-faststore-storefronts) on your production account, as described in [Step 5 - Recreating the storefront content](#step-5-recreating-the-storefront-content). Subaccount CMS content isn't carried over automatically.
4. **Verify front-end third-party tools:** Since the front-end third-party tools you reimplemented in [Step 6 - Reimplementing front-end third-party tools](#step-6-reimplementing-front-end-third-party-tools) are part of your FastStore code, they come with the connected repository. Confirm they're firing correctly on the production project.

> ℹ️ Your production account's backend apps, integrations, and Intelligent Search are already in place from your live storefront, and you don't need to recreate these settings.

### Step 9 - Adding the production domain to WebOps

> ℹ️ [DNS configuration on VTEX](https://help.vtex.com/en/docs/tutorials/how-dns-configuration-works-on-vtex) points to the platform in general and isn't exclusive to Store Framework or FastStore. Since your production domain already works on Store Framework, you don't need to configure a new domain or subdomain for the traffic split. You only need to register your existing production domain in FastStore WebOps, as described below, and share this context with [VTEX Support](https://help.vtex.com/en/support) when you request the traffic split in [step 10](#step-10-requesting-the-traffic-split), so they can confirm everything is set up as expected.

Before requesting the traffic split, add your production domain to FastStore WebOps by following these steps:

1. In your VTEX Admin, go to **Storefront > FastStore WebOps**.
2. Navigate to the **Settings** tab.
3. In the **Production domain** card, select your production domain from the dropdown list under the **Host** section.
4. Click `Add`.

    ![adding-host-webops](https://cdn.jsdelivr.net/gh/vtexdocs/dev-portal-content@main/docs/faststore/docs/go-live/adding-host-webops.png)

> ℹ️ Adding the domain doesn't change your live storefront, which keeps pointing to Store Framework until you request a traffic split.

### Step 10 - Requesting the traffic split

> ⚠️ Before requesting the traffic split, make sure your FastStore storefront is fully ready to go live. Confirm you recreated the FastStore project in your production account in [step 8](#step-8-setting-up-faststore-in-your-production-account) and that the production domain was added to WebOps in [step 9](#step-9-adding-the-production-domain-to-webops).

Once you've completed the previous steps, open a ticket with [VTEX Support](https://help.vtex.com/en/support) at least five days before your planned migration date to request traffic splitting between Store Framework and FastStore.

When the split is enabled, we recommend starting with 0% of traffic routed to FastStore and 100% kept on Store Framework for final validations on the production domain. After confirming that routing, analytics, and critical flows are working as expected, gradually increase the FastStore share. This gradual rollout allows you to slowly redirect users from the old store to the new one, run A/B tests with your analytics or testing tools, monitor performance, and ensure stability before going fully live on FastStore.

After the split is active, you can force each storefront version for QA and debugging by adding the `ab_test` query parameter to the URL:

- To force the Store Framework version: `https://www.store.com/?ab_test=v1`
- To force the FastStore version: `https://www.store.com/?ab_test=v2`

To keep the shopper experience consistent, VTEX persists the assigned storefront version in the `VtexStoreVersion` cookie, so the same shopper stays on the same variant rather than being re-randomized between the two versions on subsequent requests.

You can also read the `VtexStoreVersion` cookie in tools such as Google Tag Manager (GTM) or Google Analytics (GA) to compare metrics across the two storefront variants.

> ℹ️ By default, the `VtexStoreVersion` cookie is set as `HttpOnly`, which prevents client-side tools like GTM from reading it. If you need to read it in the browser for analytics, request this adjustment when you open the traffic split ticket.

Once you've validated the FastStore storefront, make it your live store and retire the Store Framework storefront.
